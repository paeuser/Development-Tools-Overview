Essential Software Principles

SOLID Principles

  • S - Single Responsibility Principle (SRP): A class should have only one reason to change.
  • O - Open/Closed Principle (OCP): Software entities (classes, modules, functions, etc.) should be open for extension, but closed for modification.
  • L - Liskov Substitution Principle (LSP): Objects in a program should be replaceable with instances of their subtypes without altering the correctness of that program.
  • I - Interface Segregation Principle (ISP): Clients should not be forced to depend on interfaces they do not use.
  • D - Dependency Inversion Principle (DIP): High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details. Details should depend on abstractions.

DRY (Don't Repeat Yourself)

This principle states that "Every piece of knowledge must have a single, unambiguous, authoritative representation within a system."

It aims to reduce repetition of software patterns, replacing them with abstractions or data normalization to avoid redundancy and make systems easier to maintain and extend.

KISS (Keep It Simple, Stupid)

The KISS principle advocates for simplicity in design and development. It suggests that most systems work best if they are kept simple rather than made complex.

Simplicity should be a key goal in design, and unnecessary complexity should be avoided. This often leads to more robust, maintainable, and understandable code.

YAGNI (You Aren't Gonna Need It)

This agile principle states that you should not add functionality until it's actually required.

It discourages developers from writing code that they *think* might be useful in the future but isn't needed for the current requirements. This helps prevent over-engineering and reduces wasted effort.

Separation of Concerns (SoC)

SoC is a design principle for separating a computer program into distinct sections, such that each section addresses a separate concern.

A "concern" is a set of information that affects the program's code. This leads to modularity, making it easier to develop and maintain software by dividing it into smaller, more manageable parts.

Dependency Injection (DI)

DI is a technique whereby one object supplies the dependencies of another object.

It's a way to implement Inversion of Control (IoC) for software engineering. It helps in creating loosely coupled, testable, and maintainable code by externalizing the creation and management of dependencies.

Inversion of Control (IoC)

IoC is a programming principle where the control of objects or portions of a program is transferred to a container or framework.

Instead of the application calling the framework, the framework calls the application. This makes the system more modular, extensible, and easier to test.

Clean Code

Clean code is code that is easy to understand, easy to change, and easy to extend.

It emphasizes readability, maintainability, and testability. Principles like meaningful names, small functions, clear comments (when necessary), and avoiding duplication contribute to clean code.

Test-Driven Development (TDD)

TDD is a software development process where tests are written *before* the code they are testing.

The cycle is: Red (write a failing test), Green (write just enough code to make the test pass), Refactor (improve the code while keeping tests green). This leads to higher quality code, fewer bugs, and better design.

Domain-Driven Design (DDD)

DDD is an approach to software development that centers the development on a rich understanding of the domain (the subject area of the software).

It involves creating a "Ubiquitous Language" shared by developers and domain experts, and focusing on core domain logic. This helps in building complex systems that accurately model real-world business processes.

Common Terms for Frontend Tools

Below is a list of common terms and concepts that are relevant across all these frontend tools:

Term Explanation
SPA (Single Page Application) A web application that loads a single HTML page and dynamically updates content as the user interacts with the app.
MVVM (Model-View-ViewModel) A software architectural pattern that facilitates the separation of the development of the graphical user interface.
Virtual DOM A programming concept where a virtual representation of the UI is kept in memory and synced with the real DOM.
Data Binding A technique to bind data sources to UI elements, enabling automatic updates between them.
State Management The process of managing the state of an application, often using libraries like Vuex, Redux, or Context API.
Component Lifecycle The series of methods that are invoked in different stages of a component's existence.
Routing The process of navigating between different views or pages in a web application.
Dependency Injection A design pattern used to achieve Inversion of Control (IoC) by injecting dependencies into a class.
Reactive Programming A programming paradigm oriented around data streams and the propagation of change.
Progressive Web Apps (PWA) Web applications that use modern web capabilities to deliver an app-like experience to users.
Server-Side Rendering (SSR) The process of rendering web pages on the server instead of the client.
Client-Side Rendering (CSR) The process of rendering web pages directly in the browser using JavaScript.
TypeScript A strongly typed programming language that builds on JavaScript.
RESTful APIs APIs that adhere to the REST architectural style, enabling interaction with web services.
GraphQL A query language for APIs and a runtime for executing those queries.
Testing Frameworks Tools like Jest, Mocha, and Jasmine used for testing JavaScript applications.
Build Tools Tools like Webpack, Vite, and Parcel used to bundle and optimize code for production.
Package Managers Tools like npm, yarn, and pnpm used to manage project dependencies.

Responsive Design An approach to web design that makes web pages render well on a variety of devices and screen sizes.

Pinia Feature Store Example (Vue.js)

This is an example of a Pinia store for feature management in a Vue.js application, with line-by-line explanations:

// Import Pinia's defineStore function to create a store
import { defineStore } from "pinia"; // 1
// Import Vue's reactive and ref for state management
import { reactive, ref } from "vue"; // 2
// Import axios for HTTP requests
import axios from "axios"; // 3
// Import a TypeScript type for the API response
import type { FeatureResponse } from "@/models/Responses/FeatureResponse"; // 4

// Define an interface for the feature map object
interface FeatureMap { // 5
  [key: string]: boolean;
}

// Create a Pinia store named "feature"
export const useFeatureStore = defineStore("feature", () => { // 6
  // Create a reactive object to track feature flags
  const feature: FeatureMap = reactive({ // 7
    commissions: false,
    left_nav: false,
    productioncredit: false,
    listbill: false,
    claims: false,
    taxforms: false,
    casemanagement: false,
    commissiondates: false,
  });

  // Create a ref to control navigation visibility
  const hideNav = ref(false); // 8

  // Define an async function to check if a feature is enabled
  async function checkFeature(featureName: string) { // 9
    try {
      // Make an HTTP GET request to check the feature status
      const featureResponse = await axios.get( // 10
        `/features?feature=${featureName}`,
      );

      // Update the feature flag based on the response
      feature[featureName] = featureResponse.data.isEnabled; // 11
    } catch (error) {
      // Log errors and set the feature flag to false if the request fails
      console.error(error); // 12
      feature[featureName] = false;
    }
  }

  // Return the state and actions for use in components
  return { // 13
    feature,
    hideNav,
    checkFeature,
  };
});
  • 1-4: Import necessary libraries and types for Pinia, Vue, axios, and the API response.
  • 5: Define the FeatureMap interface for feature flags.
  • 6: Create a Pinia store named feature.
  • 7: Define a reactive object to hold feature flags.
  • 8: Define a ref for navigation visibility.
  • 9-12: Define an async function to check and update feature flags from an API.
  • 13: Return the state and actions for use in Vue components.

Note: This code is for use in a Vue 3 + Pinia project, not directly in HTML. It is shown here for educational purposes.

Pinia Feature Store Sequence Diagram

The following sequence diagram illustrates the flow when checking if a feature is enabled in a Vue.js application using Pinia:

sequenceDiagram
    participant Component
    participant Store
    participant API
    Component->>Store: Call checkFeature(featureName)
    Store->>API: HTTP GET /features?feature=featureName
    API-->>Store: Respond with isEnabled
    Store-->>Store: Update feature state
    Store-->>Component: State is reactive, UI updates
  • 1. The Vue component calls checkFeature from the Pinia store when it needs to check a feature.
  • 2. The store makes an HTTP request to the API endpoint.
  • 3. The API responds with the feature status.
  • 4. The store updates its reactive state.
  • 5. The UI automatically updates based on the new state.

This sequence ensures the UI always reflects the latest feature status from the backend.

Pinia Store State Lifetime

How long does the state live?

  • Pinia store state lives as long as the page is loaded in the browser. If you navigate between pages using Vue Router (client-side navigation), the state is preserved and you do not need to call the API again.
  • If you do a full page reload (F5 or browser refresh), the state is lost and you will need to call the API again to repopulate the store.
  • If you want the state to persist across reloads, you can use a plugin like pinia-plugin-persistedstate to save the state to localStorage or sessionStorage.

Summary: As long as you stay within your SPA (Single Page Application) and do not reload the browser, the Pinia store state is kept in memory and shared across all pages.

Senior Pull Request Review Notes

Note: Pull request reviews are not about enforcing personal code style or preferred solutions.
  • Focus on scalability, supportability, and understandability for all developers, including juniors.
  • Do not reject code just because you dislike a particular style or want to rename variables.
  • Watch for overengineering and unnecessary abstractions that make code harder to support.
  • Ensure code follows the Single Responsibility Principle and is easy to test.
  • Check for edge cases, not just the happy path (e.g., failed states, error handling, network failures, unexpected backend data).
  • Review tests for meaningful coverage, not just quantity. AI-generated tests should make sense and cover real scenarios.
  • Be mindful of heavy computations and their impact.
  • Verify correct use of git and proper commit practices.
  • Ensure the pull request is named with the ticket ID.

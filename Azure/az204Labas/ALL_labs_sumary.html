Lab 1: Building Web Apps on Azure PaaS

A high-level overview of deploying a two-tier web application using Azure Platform as a Service.

Core Objective

This lab demonstrated how to create and deploy a web application consisting of a backend API and a frontend web app, leveraging Azure's Platform as a Service (PaaS) offerings. The focus was on understanding how these components work together in a cloud environment.

Key Components and Their Roles

1. Azure Storage Account

  • **Purpose:** Used for storing data, specifically images in this lab.
  • **Functionality:** Acted as the central repository for image files, accessible by the backend API.

2. Backend API (Azure Web App)

  • **Purpose:** Handled business logic and data interaction.
  • **Functionality:** Responsible for listing and uploading images to the Azure Storage Account. It was configured with an Application Setting to securely connect to storage.

3. Frontend Web Application (Azure Web App)

  • **Purpose:** Provided the user interface for interacting with the application.
  • **Functionality:** Displayed images from the backend API and allowed users to upload new ones. It was configured with an Application Setting to connect to the backend API.

Core Concepts Demonstrated

  • **Platform as a Service (PaaS):** Azure managed the underlying infrastructure (servers, operating systems, networking), allowing focus purely on application code.
  • **Application Settings:** Securely passing configuration values (like connection strings and API URLs) to deployed applications.
  • **Azure CLI Zip Deployment:** Efficiently deploying application code to Azure Web Apps using compressed packages.
  • **Two-Tier Architecture:** Understanding how a frontend application communicates with a separate backend API, which in turn interacts with data storage.

Lab 2: Implementing Task Processing with Azure Functions

A high-level overview of developing, testing, and deploying serverless functions.

Core Objective

This lab demonstrated how to create flexible, on-demand task processing logic using Azure Functions. It covered various ways functions can be triggered and how they can integrate with other Azure services.

Key Components and Their Roles

1. Azure Storage Account

  • **Purpose:** Provided persistent storage for function app internal operations and for application data (e.g., configuration files).
  • **Functionality:** Essential for Azure Functions runtime and for functions integrating with Blob storage.

2. Azure Function App

  • **Purpose:** Hosted the serverless functions in the cloud.
  • **Functionality:** Provided the execution environment, scaling, and management for the deployed functions. Configured with .NET 8 (LTS) isolated worker model on Linux.

3. Implemented Azure Functions (Examples)

  • **HTTP-Triggered Function (Echo):** Responded to web requests, demonstrating basic API functionality.
  • **Timer-Triggered Function (Recurring):** Executed automatically on a predefined schedule, ideal for background tasks.
  • **HTTP-Triggered Function with Blob Input (GetSettingInfo):** Demonstrated integration with Azure Storage by reading a configuration file from Blob storage in response to an HTTP request.

Core Concepts Demonstrated

  • **Serverless Compute (PaaS):** Focusing on code execution without managing underlying infrastructure.
  • **Azure Functions Core Tools:** Developing and testing functions locally before deployment.
  • **Function Triggers:** Understanding different ways functions can be activated (HTTP, Timer, Blob, etc.).
  • **Input/Output Bindings:** Seamlessly connecting functions to other Azure services (like Storage) without complex code.
  • **Azure CLI Deployment:** Publishing local function projects to Azure Function Apps.

Lab 3: Retrieving Azure Storage Resources with .NET SDK

A high-level overview of interacting with Azure Storage using the .NET SDK.

Core Objective

This lab focused on programmatically accessing Azure Storage containers and blobs within a C# application using the Azure Storage SDK for .NET. It demonstrated how to retrieve account metadata, enumerate containers and blobs, and obtain direct URIs to blob content.

Key Components and Their Roles

1. Azure Storage Account

  • **Purpose:** Centralized cloud storage for various data types, including blobs (files).
  • **Functionality:** Served as the target for programmatic access, holding containers and blobs with different access levels.

2. .NET Console Application (BlobManager)

  • **Purpose:** The local C# application developed to interact with Azure Storage.
  • **Functionality:** Used the Azure Storage SDK to perform operations like connecting to storage, enumerating resources, and retrieving blob URIs.

3. Azure Storage SDK for .NET

  • **Purpose:** A library that simplifies interaction with Azure Storage services from .NET applications.
  • **Functionality:** Provided classes and methods (e.g., `BlobServiceClient`, `BlobContainerClient`, `BlobClient`) to authenticate, manage, and access storage resources.

Core Concepts Demonstrated

  • **Programmatic Azure Storage Access:** Connecting to and managing Azure Storage resources using C# code.
  • **Storage Account Metadata:** Retrieving information about the storage account itself (e.g., account kind, SKU).
  • **Container and Blob Enumeration:** Listing existing containers and the blobs within them.
  • **Blob URIs and Public Access:** Understanding how to get direct web addresses for blobs and the implications of public access settings on containers.
  • **Shared Key Authentication:** Using storage account names and keys for secure access from applications.

Lab 4: Constructing a Polyglot Data Solution

A high-level overview of integrating different data stores in Azure for varied data needs.

Core Objective

This lab demonstrated how to build a "polyglot" data solution in Azure, meaning using different types of data stores (like NoSQL for flexible data and Blob Storage for files) to meet diverse application requirements. It covered data ingestion and displaying data through a web interface.

Key Components and Their Roles

1. Azure Cosmos DB (NoSQL)

  • **Purpose:** A globally distributed, multi-model database service for flexible, schema-less data (like JSON documents).
  • **Functionality:** Used to store and query product model data, demonstrating its suitability for high-performance, flexible data access.

2. Azure Storage Account (Blob Storage)

  • **Purpose:** Scalable and secure storage for unstructured data like images, videos, and files.
  • **Functionality:** Used to store image assets associated with the product data, demonstrating its role in handling binary files.

3. .NET Applications (C#)

  • **Data Upload Console App:** Programmatically ingested JSON data into Cosmos DB.
  • **Web Application:** Accessed and displayed data from both Cosmos DB and Blob Storage in a user-friendly interface.

Core Concepts Demonstrated

  • **Polyglot Persistence:** Using multiple data storage technologies (Cosmos DB for documents, Blob Storage for images) within a single solution.
  • **Azure Cosmos DB for NoSQL:** Creating accounts, databases, and containers, and uploading JSON data.
  • **Azure Storage Blob Containers:** Uploading and managing image files.
  • **Azure SDK for .NET:** Programmatically interacting with Azure Cosmos DB and Azure Storage from C# applications.
  • **Data Integration:** Connecting a web application to different Azure data stores to retrieve and display comprehensive information.
  • **Partitioning:** Understanding how partitioning keys (like `/Category` for Cosmos DB) influence data distribution and query performance.

Lab 5: Deploying Compute Workloads with Images & Containers

A high-level overview of containerizing applications and deploying them to various Azure container services.

Core Objective

This lab demonstrated how to containerize a .NET application using Docker and deploy it to different Azure container services, including Azure Container Registry, Azure Container Instances, and Azure Container Apps. The focus was on understanding the lifecycle of containerized applications in Azure.

Key Components and Their Roles

1. Docker & Dockerfile

  • **Purpose:** To package the .NET application and its dependencies into a standardized, portable container image.
  • **Functionality:** The Dockerfile provided instructions for building the image (e.g., base image, working directory, copying files, publishing, entry point).

2. Azure Container Registry (ACR)

  • **Purpose:** A private, managed Docker registry in Azure for storing and managing container images.
  • **Functionality:** Acted as the central repository where the built Docker image was pushed and stored.

3. Azure Container Instances (ACI)

  • **Purpose:** A service for running containers directly in Azure without managing virtual machines or orchestration.
  • **Functionality:** Used to quickly deploy and run the containerized .NET application for task-based workloads, demonstrating both automatic and manual deployment methods.

4. Azure Container Apps

  • **Purpose:** A serverless platform for building and deploying modern apps and microservices using containers.
  • **Functionality:** Demonstrated deploying a sample container app to a Container Apps environment, showcasing its capabilities for hosting web applications and APIs.

Core Concepts Demonstrated

  • **Containerization:** Packaging applications and their dependencies into portable Docker images.
  • **Container Image Management:** Storing and retrieving container images from a private registry (ACR).
  • **Ephemeral Compute:** Running short-lived, task-based containers (ACI).
  • **Serverless Containers:** Deploying and scaling containerized applications in a managed, serverless environment (Azure Container Apps).
  • **Azure CLI for Containers:** Using command-line tools to build, push, and deploy containers across various Azure services.

Lab 6: Authenticating with OpenID Connect, MSAL, & .NET SDKs

A high-level overview of securing applications using Microsoft Entra ID for identity management.

Core Objective

This lab demonstrated how to secure a web application by integrating it with Microsoft Entra ID (formerly Azure Active Directory) for user authentication. It focused on using OpenID Connect (OIDC) and the Microsoft Authentication Library (MSAL) with .NET SDKs to manage user access.

Key Components and Their Roles

1. Microsoft Entra ID

  • **Purpose:** Microsoft's cloud-based identity and access management service.
  • **Functionality:** Used to register the web application, manage user accounts, and provide the authentication service for the application.
  • **When to Use:** When you need a centralized identity provider for your applications, manage user access, or integrate with other Microsoft cloud services.

2. ASP.NET Web App (OIDCClient)

  • **Purpose:** The client application that requires user authentication.
  • **Functionality:** Configured to interact with Microsoft Entra ID using OpenID Connect to sign in users and manage their sessions.
  • **When to Use:** For building web applications (e.g., MVC, Razor Pages) where users need to log in securely using Microsoft identities.

3. OpenID Connect (OIDC) & MSAL .NET SDKs

  • **Purpose:** OIDC is an authentication protocol built on OAuth 2.0; MSAL is Microsoft's library for integrating with Microsoft identity platform.
  • **Functionality:** These technologies facilitate the secure flow of authentication tokens between the web app and Microsoft Entra ID, allowing the app to verify user identities.
  • **When to Use:** When implementing modern authentication flows for applications interacting with Microsoft Entra ID or other identity providers that support OIDC.

Core Concepts Demonstrated

  • **Application Registration:** Registering an application in Microsoft Entra ID to define its identity and permissions.
  • **Redirect URIs:** Configuring URLs where Microsoft Entra ID sends authentication responses after a user logs in.
  • **Single-Tenant Authentication:** Securing an application to allow access only to users within a specific Microsoft Entra ID directory.
  • **User Principal Name (UPN):** Understanding the format and use of user identities in Microsoft Entra ID.
  • **Local Development with HTTPS:** Setting up and trusting self-signed certificates for secure local testing of authentication flows.
  • **`Microsoft.Identity.Web` Library:** Simplifying the integration of Microsoft identity platform authentication into ASP.NET Core applications.
